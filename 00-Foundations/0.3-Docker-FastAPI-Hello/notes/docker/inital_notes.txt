# Docker Foundations Notes

---

# 1. What Problem Docker Solves

Before Docker:

```text id="d08yy4"
"Works on my machine"
```

problem happened constantly.

Different machines had:

* different Python versions
* missing packages
* OS differences
* dependency conflicts

Docker solves this by packaging:

* code
* dependencies
* runtime
* environment
* startup instructions

into one portable unit.

---

# 2. Core Docker Concepts

---

# Docker Image

## Definition

A Docker image is:

```text id="l0ehjx"
A read-only blueprint/template used to create containers.
```

It contains:

* OS layers
* Python/runtime
* libraries
* dependencies
* app code
* startup instructions

---

# Mental Models

| Real World        | Docker |
| ----------------- | ------ |
| Cake recipe       | Image  |
| Blueprint         | Image  |
| Game installation | Image  |
| Class definition  | Image  |

---

# Important Properties

| Property   | Value |
| ---------- | ----- |
| Alive?     | NO    |
| Executing? | NO    |
| Stored?    | YES   |
| Reusable?  | YES   |

---

# Docker Container

## Definition

A container is:

```text id="mjlwmk"
A running instance created from an image.
```

Container includes:

* running processes
* memory
* filesystem
* networking
* ports
* runtime state

---

# Mental Models

| Real World           | Docker    |
| -------------------- | --------- |
| Actual cake          | Container |
| Running robot        | Container |
| Running game session | Container |
| Object from class    | Container |

---

# Image vs Container

| Image          | Container              |
| -------------- | ---------------------- |
| Blueprint      | Running instance       |
| Passive        | Active                 |
| Stored         | Executing              |
| Reusable       | Temporary runtime      |
| Immutable base | Writable runtime layer |

```text id="gtjlwm"
Docker Client

      ↓

      ↓
```


* faster rebuilds

# 5. Running a Container

---

# Syntax

```bash id="jlwm23"
docker run -it IMAGE_NAME
```

Example:

```bash id="jlwm66"
docker run -it python:3.11-slim
```

---

# Flag Meaning

| Flag | Meaning          |
| ---- | ---------------- |
| -i   | interactive mode |
| -t   | terminal access  |

Combined:

```bash id="jlwm11"
-it
```

means:

```text id="jlwm78"
Attach interactive terminal to container.
```

---

# What Happens Internally

```text id="jlwm90"
1. Docker takes image
2. Creates writable layer
3. Allocates resources
4. Creates isolated environment
5. Starts process
6. Container becomes alive
```

# 7. Temporary Nature of Containers

Containers are designed to be:

```text id="jlwm71"
Disposable and replaceable
```

---

# Writable Layer

Containers have temporary writable storage.

Example:

Inside container:

```bash id="jlwm72"
touch hello.txt
```

If container gets removed:

```bash id="jlwm73"
docker rm container_id
```

then:

```text id="jlwm74"
hello.txt disappears
```

---

# Why Docker Works This Way

Modern cloud philosophy:

```text id="jlwm75"
If service breaks,
destroy and recreate.
```

NOT:

```text id="jlwm76"
Repair machine manually forever.
```

---

# Persistent Storage

Permanent data is stored using:

```text id="jlwm77"
Volumes
```

Volumes survive container deletion.

---

# 8. Isolation in Containers

Containers behave like:

```text id="jlwm79"
Tiny isolated computers
```

Each container gets isolated:

* filesystem
* processes
* networking
* ports
* environment

---

# Why Isolation Matters

Benefits:

* reproducibility
* security
* scalability
* dependency separation

# 10. Dockerfile Basics

A Dockerfile defines how to build image.

---

# Basic Syntax

```dockerfile id="jlwm81"
FROM python:3.11-slim

WORKDIR /app

COPY . .

RUN pip install -r requirements.txt

CMD ["python", "main.py"]
```

---

# Dockerfile Instruction Breakdown

---

## FROM

```dockerfile id="jlwm82"
FROM python:3.11-slim
```

Base image.

---

## WORKDIR

```dockerfile id="jlwm83"
WORKDIR /app
```

Sets working directory inside container.

Equivalent to:

```bash id="jlwm84"
cd /app
```

---

## COPY

```dockerfile id="jlwm85"
COPY . .
```

Copies files into container.

---

## RUN

```dockerfile id="jlwm86"
RUN pip install -r requirements.txt
```

Runs command during image build.

---

## CMD

```dockerfile id="jlwm87"
CMD ["python", "main.py"]
```

Default startup command.

Runs when container starts.

# 13. Good Practices

---

# Use Slim Images

Prefer:

```text id="jlwm95"
python:3.11-slim
```

instead of huge images.

Benefits:

* faster builds
* smaller size
* lower attack surface

---

# Keep Containers Stateless

Do NOT rely on container internal storage.

Use:

* databases
* volumes
* cloud storage

---

# Rebuild Instead of Manual Fixes

Good Docker mindset:

```text id="jlwm96"
Destroy and recreate cleanly.
```

NOT:

```text id="jlwm97"
Modify containers manually forever.
```

---

# Use Version Tags

GOOD:

```text id="jlwm98"
python:3.11-slim
```

BAD:

```text id="jlwm99"
python:latest
```

Reason:

* reproducibility
* predictable environments

---

# Keep Images Small

Avoid unnecessary packages.

Smaller images:

* deploy faster
* start faster
* cost less

---

# 14. Important Production Mindset

Containers are NOT virtual machines.

Containers share host kernel but isolate processes.

Containers are:

```text id="jlwm100"
Lightweight isolated runtime environments.
```
